## 1. Por qué un contrato de evento explícito

Los consumidores deben poder:

- **identificar** el mensaje de forma única (`eventId`);
- **enrutar** por tipo (`eventType`);
- **interpretar** el cuerpo (`payload`) sin acoplar al modelo interno del productor.

En Arkano el contrato común es el **EventEnvelope** + documentación en [event-contracts.md](../../03-event-driven/event-contracts.md).

---

## 2. Envelope: campos obligatorios

Definición TypeScript compartida por convención en cada servicio (ejemplo **accounts-service**):

```1:18:services/accounts-service/src/common/events/event-envelope.ts
export const EVENT_VERSION = 1;

export type EventEnvelope<T = Record<string, unknown>> = {
  eventId: string;
  eventType: string;
  source: string;
  occurredAt: string;
  version: number;
  payload: T;
};

export function parseEnvelope(raw: string): EventEnvelope {
  const v = JSON.parse(raw) as EventEnvelope;
  if (!v?.eventId || !v?.eventType || !v?.source) {
    throw new Error('Invalid event envelope');
  }
  return v;
}
```

- **`eventId`:** UUID por **ocurrencia** de mensaje (nuevo en cada publicación desde outbox con `randomUUID()`).
- **`source`:** nombre lógico del servicio emisor (`accounts-service`, `transactions-service`).
- **`version`:** hoy `1`; reservado para cambios de esquema del envelope o semántica.

---

## 3. Validación al consumir (`parseEnvelope`)

Los consumidores Kafka convierten `message.value` a string, parsean JSON y delegan en `parseEnvelope`. Si falla, en **ai-service** se hace skip con warning (mensaje inválido no entra al pipeline de negocio).

## 4. Topics y constantes

Ejemplo **accounts** (mismo concepto en **transactions**; **ai** usa `transaction-events` + DLQ):

```1:3:services/accounts-service/src/common/topics.ts
export const TOPIC_ACCOUNT_EVENTS = 'account-events';
export const TOPIC_TRANSACTION_EVENTS = 'transaction-events';
```

**ai-service** añade:

```1:2:services/ai-service/src/common/topics.ts
export const TOPIC_TRANSACTION_EVENTS = 'transaction-events';
export const TOPIC_TRANSACTION_EVENTS_DLQ = 'transaction-events-dlq';
```

La tabla **outbox** almacena el nombre del topic en columna `topic` para que el publicador sea genérico.

## 5. Tipos de evento en el sistema

Resumen alineado con [event-contracts.md](../../03-event-driven/event-contracts.md):

| `eventType` | Publicado por | Rol |
|-------------|---------------|-----|
| `ClientCreated` | accounts | Alta de cliente |
| `AccountCreated` | accounts | Nueva cuenta + saldo inicial en payload |
| `BalanceUpdated` | accounts | Saldo autoritativo tras `TransactionCompleted` |
| `TransactionRequested` | transactions | Solicitud asíncrona de ejecución |
| `TransactionCompleted` | transactions | Pata contable aplicable en accounts (una o más por transferencia) |
| `TransactionRejected` | transactions | Motivo de rechazo |

Los consumidores filtran por `eventType` antes de aplicar lógica (ej. accounts solo procesa `TransactionCompleted` en `transaction-events`).

---

## 6. Diseño de payload y granularidad

- **`TransactionCompleted`:** incluye `transactionId`, `accountId`, `amount` (con signo según pata). Una **transferencia** genera **dos** eventos con **dos** `eventId` distintos — diseño explícito para partición por cuenta e idempotencia por pata (ver [Idempotencia §6](./3.Idempotencia.md#6-caso-2-accounts-service--doble-capa-eventid--patas-aplicadas)).
- **`TransactionRequested`:** incluye tipo, importe y cuentas opcionales según tipo; el ejecutor valida contra snapshots.

Los payloads se serializan como JSON dentro del envelope en la columna `payload` del outbox.

---

## 7. Particionado y orden

- **`partitionKey`** en outbox suele ser **`accountId`** (o equivalente) para que eventos de la **misma cuenta** conserven **orden relativo** dentro de la partición.
- **Múltiples particiones** en `transaction-events` (p. ej. 3 en ai-service al crear topics) permiten paralelismo; el orden **global** entre cuentas no está garantizado.

---

## 8. Evolución y versionado

- Incrementar **`EVENT_VERSION`** cuando cambie el contrato del envelope o la semántica obligatoria.
- Para payloads: preferir **campos opcionales** nuevos o nuevos `eventType` antes de romper consumidores existentes.
- Documentar cambios en `docs/03-event-driven/event-contracts.md` y, si aplica, en esta serie de fundamentos.

## 9. Glosario

| Término | Definición |
|---------|------------|
| **Envelope** | Contenedor común alrededor del payload de negocio |
| **Event type** | Nombre semántico del suceso (`TransactionCompleted`, …) |
| **Partition key** | Clave usada por Kafka para elegir partición |
