## 1. Qué es la idempotencia

Una operación es **idempotente** si ejecutarla **más de una vez** produce el **mismo efecto observable** que ejecutarla **una sola vez** (salvo tal vez logs o timestamps de auditoría).

En banca y en sistemas con dinero, la idempotencia evita:

- **doble aplicación** de un mismo movimiento al saldo;
- **doble publicación** de efectos irreversibles;
- **inconsistencias** cuando el mensajería reentrega mensajes.

No confundir con “solo ejecutar una vez a nivel de red”: en la práctica **sí** puede llegar el mismo evento dos veces; la idempotencia está en **tu código y tu modelo de datos**.

---

## 2. Por qué Kafka y la idempotencia van juntos

Kafka (y clientes típicos) suelen ofrecer semántica **at-least-once**: el broker puede considerar entregado un mensaje y el consumidor puede **caer antes de comprometer el offset**, o haber **reintentos** en el productor. Resultado: **el mismo registro puede procesarse otra vez**.

Por tanto, el consumidor debe diseñarse para que:

- la **segunda** ejecución con el mismo significado de negocio **no duplique** efectos, o
- use un **registro previo** (“ya procesé este `eventId`”) y salga sin mutar de nuevo.

En Arkano eso se documenta también en [idempotency-strategy.md](../../03-event-driven/idempotency-strategy.md) y en la guía [Kafka](./2.kafka.md) (garantías de entrega).

## 3. Claves naturales frente a identificador de evento

- **`eventId` (en el envelope):** identifica **esta ocurrencia concreta** del mensaje en el bus. Dos publicaciones distintas del “mismo hecho” desde el punto de vista de negocio **deberían** llevar dos `eventId` distintos si son dos eventos distintos; un **reenvío** del broker del **mismo** mensaje conserva el mismo `eventId`.

- **Clave de negocio compuesta:** p. ej. `(transactionId, accountId)` para una **pata** de `TransactionCompleted`: el mismo movimiento puede generar **dos** mensajes Kafka (origen y destino en una transferencia), cada uno con su **`eventId`** pero compartiendo `transactionId` y distinto `accountId` / `amount`.

En **accounts-service**, la tabla `applied_transaction_legs` implementa idempotencia por **pata**; `processed_events` por **`eventId` del mensaje Kafka**.

## 4. Patrón `processed_events` en Arkano

Cada microservicio tiene **su propia base de datos** y **su propia tabla** `processed_events` (mismo concepto, sin compartir filas entre servicios).

Definición TypeORM (ejemplo en accounts; es equivalente en transactions y ai):

```1:10:services/accounts-service/src/infrastructure/persistence/processed-event.orm-entity.ts
import { Column, Entity, PrimaryColumn } from 'typeorm';

@Entity('processed_events')
export class ProcessedEventOrmEntity {
  @PrimaryColumn({ name: 'event_id', type: 'uuid' })
  eventId!: string;

  @Column({ name: 'processed_at', type: 'timestamptz' })
  processedAt!: Date;
}
```

**Semántica:** si existe una fila con `event_id = env.eventId`, el consumidor **ya tomó una decisión final** para ese mensaje (aplicar efectos, o marcar como procesado sin efecto, p. ej. cuenta desconocida).

## 5. Caso 1: transactions-service — `TransactionExecuteService`

**Responsabilidad:** procesar el evento **`TransactionRequested`** y pasar la transacción a `completed` / `rejected`, escribiendo nuevas filas de outbox (`TransactionCompleted` / `TransactionRejected`).

**Idempotencia por `eventId`:** al entrar en la transacción de aplicación, si el `eventId` del envelope ya está en `processed_events`, **sale sin hacer nada**:

```29:36:services/transactions-service/src/infrastructure/kafka/transaction-execute.service.ts
    await this.dataSource.transaction(async (em) => {
      const procRepo = em.getRepository(ProcessedEventOrmEntity);
      const txnRepo = em.getRepository(TransactionOrmEntity);
      const snapRepo = em.getRepository(AccountSnapshotOrmEntity);
      const outboxRepo = em.getRepository(OutboxEventOrmEntity);

      if (await procRepo.findOne({ where: { eventId: env.eventId } })) return;
```

**Otro cierre idempotente:** si la transacción ya no está en `pending` (p. ej. ya fue procesada por un intento anterior que guardó estado pero el offset no se comprometió igual), solo se **persiste** el `eventId` como procesado y no se vuelve a ejecutar la lógica de negocio:

```39:46:services/transactions-service/src/infrastructure/kafka/transaction-execute.service.ts
      const txn = await txnRepo.findOne({ where: { id: transactionId } });
      if (!txn || txn.status !== 'pending') {
        await procRepo.save({
          eventId: env.eventId,
          processedAt: new Date(),
        });
        return;
      }
```

**Por qué es un caso real:** un mismo `TransactionRequested` podría entregarse dos veces; la segunda vez no debe volver a completar ni a publicar duplicados de resultado.

## 6. Caso 2: accounts-service — doble capa (`eventId` + patas aplicadas)

**Responsabilidad:** aplicar **`TransactionCompleted`** al saldo autoritativo y encolar **`BalanceUpdated`**.

### Capa A — fuera de la transacción principal

Evita abrir transacción innecesaria si el mensaje ya se consideró procesado:

```35:39:services/accounts-service/src/infrastructure/kafka/transaction-completed-applier.service.ts
  async apply(env: EventEnvelope<TransactionCompletedPayload>): Promise<void> {
    const existing = await this.processed.findOne({
      where: { eventId: env.eventId },
    });
    if (existing) return;
```

### Capa B — dentro de la misma transacción SQL

Doble comprobación por **condiciones de carrera** entre hilos o reentradas:

```48:51:services/accounts-service/src/infrastructure/kafka/transaction-completed-applier.service.ts
      const dupProc = await procRepo.findOne({
        where: { eventId: env.eventId },
      });
      if (dupProc) return;
```

### Capa C — idempotencia de negocio por pata `(transactionId, accountId)`

Un mismo `transactionId` puede tener **varios** `TransactionCompleted` con **`eventId` distintos** (transferencia: dos mensajes). La tabla **`applied_transaction_legs`** evita aplicar **dos veces** la misma pata aunque se repitiera el mismo mensaje o hubiera colisiones de diseño:

```53:62:services/accounts-service/src/infrastructure/kafka/transaction-completed-applier.service.ts
      const leg = await legRepo.findOne({
        where: { transactionId, accountId },
      });
      if (leg) {
        await procRepo.save({
          eventId: env.eventId,
          processedAt: new Date(),
        });
        return;
      }
```

Si la pata ya existía, **no** se modifica el saldo otra vez; solo se marca este **`eventId`** como procesado (el mensaje duplicado se “absorbe”).

El resto del flujo (cuenta inexistente, saldo negativo) también termina en **`procRepo.save`** con el `eventId`, de modo que **reprocesar** el mismo evento no cambia el mundo dos veces.

## 7. Caso 3: transactions-service — snapshots (`AccountEventApplierService`)

**Responsabilidad:** mantener `account_snapshots` al día con `AccountCreated` y `BalanceUpdated` publicados por accounts.

**Patrón:** comprobación **antes** de la transacción y **dentro** de ella, igual que en accounts para `TransactionCompleted`.

Ejemplo `AccountCreated`:

```31:43:services/transactions-service/src/infrastructure/kafka/account-event-applier.service.ts
  async applyAccountCreated(
    env: EventEnvelope<AccountCreatedPayload>,
  ): Promise<void> {
    const seen = await this.processed.findOne({
      where: { eventId: env.eventId },
    });
    if (seen) return;

    const { accountId, clientId, balance } = env.payload;
    await this.snapshots.manager.transaction(async (em) => {
      const procRepo = em.getRepository(ProcessedEventOrmEntity);
      const snapRepo = em.getRepository(AccountSnapshotOrmEntity);
      if (await procRepo.findOne({ where: { eventId: env.eventId } })) return;
```

`BalanceUpdated` repite el mismo esquema (`applyBalanceUpdated`).

## 8. Caso 4: ai-service — consumidor + applier + restricción única

### 8.1 Filtro temprano en el consumidor

Antes de llamar al LLM (costoso), si `eventId` ya está en `processed_events`, se omite:

```72:79:services/ai-service/src/infrastructure/kafka/transaction-events.consumer.ts
        const existing = await this.processed.findOne({
          where: { eventId: env.eventId },
        });
        if (existing) {
          this.logger.log(
            `[EVENT-BUS] SKIP [ai-service] eventId=${env.eventId} (ya en processed_events, idempotencia)`,
          );
          return;
        }
```

Eso evita **repetir** llamadas a Ollama/mock cuando el mensaje se reentrega y el applier ya persistió en un intento anterior.

### 8.2 Dentro del applier — misma transacción que explicación + processed

Tras generar el texto con el orquestador LLM, en una transacción:

1. Si `eventId` ya está en `processed_events`, **return** (no inserta otra explicación).
2. Inserta `transaction_explanations`.
3. Inserta `processed_events`.

```38:56:services/ai-service/src/infrastructure/kafka/ai-transaction-event-applier.service.ts
    await this.explanations.manager.transaction(async (em) => {
      const procRepo = em.getRepository(ProcessedEventOrmEntity);
      const expRepo = em.getRepository(TransactionExplanationOrmEntity);

      if (await procRepo.findOne({ where: { eventId: env.eventId } })) return;

      await expRepo.save({
        transactionId,
        accountId,
        eventType: env.eventType,
        explanation,
        sourceEventId: env.eventId,
        createdAt: new Date(),
      });

      await procRepo.save({
        eventId: env.eventId,
        processedAt: new Date(),
      });
    });
```

### 8.3 Garantía en base de datos: `source_event_id` único

Aunque hubiera una condición de carrera, la columna **`source_event_id`** es **única** a nivel de tabla: un segundo `save` con el mismo `env.eventId` fallaría a nivel SQL.

```23:24:services/ai-service/src/infrastructure/persistence/transaction-explanation.orm-entity.ts
  @Column({ name: 'source_event_id', type: 'uuid', unique: true })
  sourceEventId!: string;
```

Así se combina **idempotencia lógica** (comprobar antes) con **unicidad física** (última línea de defensa).

## 9. Caso límite: DLQ en ai-service sin fila de explicación

Si tras **tres** intentos el applier sigue fallando (p. ej. Ollama caído), el consumidor envía el mensaje al **DLQ** y **guarda** el `eventId` en `processed_events` para no bloquear el topic en bucle.

En ese escenario **no** hay fila en `transaction_explanations` para ese `eventId`, pero el sistema considera el evento **cerrado** desde el punto de vista del consumidor. Es un diseño consciente (operación **no idempotente** en el sentido “mismo resultado que éxito”, pero **sí** estable: no reintentar infinito).

```104:111:services/ai-service/src/infrastructure/kafka/transaction-events.consumer.ts
        this.logger.error(
          `Moving event ${env.eventId} to DLQ after ${MAX_RETRIES} failures: ${lastError}`,
        );
        await this.kafkaService.sendDlq(env.eventId, raw);
        await this.processed.save({
          eventId: env.eventId,
          processedAt: new Date(),
        });
```

## 10. Tabla resumen por servicio

| Servicio | Tabla / mecanismo | Qué identifica | Efecto si ya existe |
|----------|-------------------|----------------|---------------------|
| **transactions** | `processed_events` | `eventId` del `TransactionRequested` | No reejecuta completar/rechazar |
| **transactions** | Estado `transactions.status` | `pending` vs otro | Marca solo `processed` si ya no aplica |
| **accounts** | `processed_events` | `eventId` de cada mensaje Kafka | Salida temprana / dentro de TX |
| **accounts** | `applied_transaction_legs` | `(transactionId, accountId)` | No vuelve a sumar saldo por la misma pata |
| **transactions** (snapshots) | `processed_events` | `eventId` de `AccountCreated` / `BalanceUpdated` | No duplica snapshot |
| **ai** | `processed_events` | `eventId` | No segunda llamada LLM / no segunda fila lógica |
| **ai** | `transaction_explanations.source_event_id` UNIQUE | `eventId` | Fallo DB si duplicado |

## 12. Buenas prácticas y errores habituales

1. **Usar el `eventId` del envelope**, no solo el `transactionId`, como única clave de deduplicación global: una transferencia tiene **varios** eventos completados con distinto `eventId`.
2. **Marcar procesado** también en ramas de error de negocio (cuenta desconocida, saldo negativo), para no reintentar eternamente el mismo mensaje mal formado o inválido para tu dominio.
3. **Transacción única** que agrupe efecto de negocio + `processed_events` (o outbox) cuando sea posible, para alinear con “exactly-once efectivo” dentro de la BD.
4. **No** confiar solo en el offset de Kafka: el offset confirma lectura; tu regla de negocio vive en **tablas de aplicación**.
5. **Productor idempotente** de Kafka (feature avanzada del cliente Java) **no sustituye** idempotencia en consumidor en este diseño Node/KafkaJS.

## 13. Glosario

| Término | En Arkano |
|---------|-----------|
| **Idempotencia** | Reprocesar el mismo mensaje no duplica efectos monetarios ni explicaciones |
| **`eventId`** | UUID del envelope; clave principal en `processed_events` |
| **Pata** | Par `(transactionId, accountId)` aplicado en accounts para un `TransactionCompleted` |
| **At-least-once** | Kafka puede reentregar; el código asume duplicados posibles |
| **DLQ** | Topic de fallos; en ai puede cerrar `processed` sin explicación |
