# 1. Qué es Event-Driven Architecture

Event-Driven Architecture (EDA) es un estilo de arquitectura donde **los servicios reaccionan a eventos**.

Un **evento** es simplemente algo que **ya ocurrió en el sistema**.

Ejemplos:

* `ClientCreated`
* `AccountCreated`
* `TransactionCompleted`
* `TransactionRejected`

En lugar de que un servicio **llame directamente a otro**, publica un evento.

Otros servicios **escuchan ese evento** y reaccionan.

---

# 2. Ejemplo simple sin arquitectura de eventos

Imaginemos un sistema bancario simple.

Cuando se crea una cuenta:

```
Client → Accounts Service → Transactions Service → AI Service
```

Flujo:

1. Accounts Service crea cuenta
2. Accounts Service llama HTTP a Transactions Service
3. Transactions Service llama HTTP a AI Service

Problemas:

* alto acoplamiento
* dependencia entre servicios
* errores en cascada

Si Transactions falla → todo falla.

---

# 3. Ejemplo con Event-Driven Architecture

Ahora usamos eventos.

```
Accounts Service → Kafka → Transactions Service
                               ↓
                            AI Service
```

Flujo:

1. Accounts Service crea cuenta
2. Publica evento `AccountCreated`
3. Transactions Service recibe evento
4. AI Service también puede reaccionar

Ningún servicio depende directamente de otro.

---

# 4. Conceptos fundamentales

## Evento

Algo que ocurrió en el sistema.

Ejemplo:

```
TransactionCompleted
```

---

## Producer

Servicio que **emite el evento**.

En tu proyecto:

```
Accounts Service
Transactions Service
```

---

## Consumer

Servicio que **escucha eventos**.

En tu proyecto:

```
Transactions Service
AI Service
```

---

## Event Broker

Sistema que transporta eventos.

En tu proyecto:

```
Kafka (Redpanda)
```

---

## Topic

Canal donde se publican eventos.

Ejemplo:

```
account-events
transaction-events
```

# 7. Event Envelope

Todos los eventos siguen una estructura estándar.

Ejemplo:

```json
{
  "eventId": "uuid",
  "eventType": "TransactionCompleted",
  "source": "transactions-service",
  "occurredAt": "2026-01-01T10:00:00Z",
  "payload": {
    "accountId": "123",
    "amount": 100
  }
}
```

Campos:

| Campo      | Descripción                |
| ---------- | -------------------------- |
| eventId    | identificador único        |
| eventType  | tipo de evento             |
| source     | servicio que generó evento |
| occurredAt | timestamp                  |
| payload    | datos del evento           |

# 8. Ventajas de Event-Driven Architecture

## Bajo acoplamiento

Los servicios no se llaman entre sí.

---

## Escalabilidad

Podemos agregar más consumidores.

---

## Resiliencia

Si un servicio falla, otros siguen funcionando.

---

## Procesamiento asíncrono

No bloquea al usuario.

---

# 9. Desventajas

EDA también tiene desafíos.

### Mayor complejidad

Sistemas distribuidos son más complejos.

---

### Consistencia eventual

Los servicios pueden tardar en sincronizarse.

---

### Debugging más difícil

Eventos viajan entre servicios.

---

### Qué es un event broker

Sistema que transporta eventos entre servicios.

Ejemplo:

```
Kafka
```

---

### Qué es un event producer

Servicio que publica eventos.

---

Servicio que escucha eventos.
