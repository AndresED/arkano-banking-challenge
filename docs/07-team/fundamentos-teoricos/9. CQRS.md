## 1. Qué es CQRS

**CQRS** separa:

- **Commands:** intenciones que **cambian** el estado del sistema (crear cuenta, solicitar transacción).
- **Queries:** lecturas que **no** deben tener efectos colaterales de negocio (obtener cuenta por id).

Cada comando o query tiene un **handler** dedicado. Eso clarifica responsabilidades y facilita pruebas unitarias por caso de uso.

---

## 2. Por qué encaja con APIs y eventos

- En **HTTP**, los verbos y rutas mapean de forma natural: `POST` → comando, `GET` → query.
- En **event-driven**, los **eventos** son hechos ya ocurridos; el “comando” suele ser la **API síncrona** que dispara el proceso (p. ej. `RequestTransaction`), mientras los consumidores Kafka aplican **proyecciones** o políticas (más cercano a “side effect” que a CQRS clásico del bus).

Arkano usa CQRS de forma **explícita en la capa HTTP** de accounts y transactions.

---

## 3. NestJS: CommandBus y QueryBus

El paquete `@nestjs/cqrs` proporciona:

- **`CommandBus.execute(command)`** → despacha al `CommandHandler` registrado.
- **`QueryBus.execute(query)`** → despacha al `QueryHandler` registrado.

El controlador solo construye el objeto comando/query y delega (ver [Hexagonal §4](./8.%20Hexagonal%20architecture.md#4-adaptadores-de-entrada-rest-y-kafka)).

Ejemplo **solicitud de transacción**: el handler implementa `ICommandHandler` y lleva la anotación `@CommandHandler(RequestTransactionCommand)`.

Dentro de `execute` se valida el DTO, se abre transacción TypeORM y se persiste negocio + outbox (ver [Outbox §4.2](./4.%20Outbox%20Pattern.md#42-solicitud-de-transacción-transactionrequested)).

**Crear cuenta** en accounts sigue el mismo patrón con `CreateAccountCommand` / `CreateAccountHandler`.

## 5. Queries en Arkano

Ejemplo **obtener cuenta por id**: el handler implementa `IQueryHandler` con tipo de retorno explícito `AccountView`.

```14:32:services/accounts-service/src/modules/accounts/application/queries/get-account-by-id.handler.ts
@QueryHandler(GetAccountByIdQuery)
export class GetAccountByIdHandler implements IQueryHandler<
  GetAccountByIdQuery,
  AccountView
> {
  constructor(
    @InjectRepository(AccountOrmEntity)
    private readonly accounts: Repository<AccountOrmEntity>,
  ) {}

  async execute(query: GetAccountByIdQuery): Promise<AccountView> {
    const row = await this.accounts.findOne({ where: { id: query.accountId } });
    if (!row) throw new NotFoundException('Account not found');
    return {
      accountId: row.id,
      clientId: row.clientId,
      balance: Number(row.balance),
    };
  }
}
```

La query **lee** la entidad ORM y devuelve una **vista** (DTO de salida conceptual), sin publicar eventos.

## 6. Registro de handlers en el módulo

En `AccountsModule`, los handlers se agrupan y se pasan a `providers`:

```18:22:services/accounts-service/src/modules/accounts/accounts.module.ts
const Handlers = [
  CreateClientHandler,
  CreateAccountHandler,
  GetAccountByIdHandler,
];
```

`CqrsModule` debe estar en `imports` para que el bus resuelva los handlers.

---

## 7. CQRS “estricto” vs uso pragmático aquí

- **No hay** modelos de lectura separados (read models) ni bases de datos distintas para read/write: **CQRS ligero**.
- Los **consumidores Kafka** no usan `CommandBus`; invocan servicios (`TransactionExecuteService`, appliers) directamente: es válido y común en microservicios.
- Un CQRS más avanzado introduciría **proyecciones** alimentadas por eventos solo en el lado lectura; aquí los **snapshots** en transactions cumplen un rol parecido pero persistidos como tabla técnica.

## 8. Glosario

| Término | Significado |
|---------|-------------|
| **Command** | Mensaje imperativo con intención de mutación |
| **Query** | Solicitud de datos sin mutación |
| **Handler** | Clase que ejecuta un command o query concreto |
