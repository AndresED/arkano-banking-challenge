## 1. Por qué falla el procesamiento de un mensaje

Causas habituales:

- **Infraestructura:** base de datos no disponible, timeout de red.
- **Dependencias externas:** en **ai-service**, fallo de **Ollama** o del mock LLM.
- **Datos:** payload inválido (en Arkano muchos casos se filtran con `parseEnvelope` y se hace `return` sin reintentar infinito).
- **Bugs:** excepciones no controladas.

La política debe equilibrar **resiliencia** (reintentos) con **no bloquear** la partición indefinidamente (DLQ o descarte documentado).

## 2. Política genérica: reintentos acotados

Patrón recomendado en el proyecto (documentado en inglés en [retry-dlq-strategy.md](../../03-event-driven/retry-dlq-strategy.md)):

- **`maxRetries`:** 3 (orden de magnitud estándar).
- **Backoff:** espera creciente entre intentos para no martillar DB o LLM.

## 3. DLQ: propósito y contrato

Un **topic DLQ** recibe mensajes que **no pudieron procesarse** tras los reintentos. Sirve para:

- inspección manual o con herramientas;
- reproceso controlado;
- no atascar el consumo del topic principal.

En Arkano, el topic se define en **ai-service** junto al topic principal:

```31:45:services/ai-service/src/infrastructure/kafka/kafka.service.ts
      await admin.createTopics({
        topics: [
          {
            topic: TOPIC_TRANSACTION_EVENTS,
            numPartitions: 3,
            replicationFactor: 1,
          },
          {
            topic: TOPIC_TRANSACTION_EVENTS_DLQ,
            numPartitions: 1,
            replicationFactor: 1,
          },
        ],
        waitForLeaders: true,
      });
```

La publicación al DLQ reutiliza el **mismo payload crudo** (`raw`) del mensaje consumido:

```63:78:services/ai-service/src/infrastructure/kafka/kafka.service.ts
  async sendDlq(key: string | null, value: string): Promise<void> {
    try {
      const j = JSON.parse(value) as { eventType?: string; eventId?: string };
      this.logger.warn(
        `[EVENT-BUS] PUBLISH [ai-service] -> topic=${TOPIC_TRANSACTION_EVENTS_DLQ} ` +
          `eventType=${j.eventType ?? '?'} eventId=${j.eventId ?? '?'} (DLQ)`,
      );
    } catch {
      this.logger.warn(
        `[EVENT-BUS] PUBLISH [ai-service] -> topic=${TOPIC_TRANSACTION_EVENTS_DLQ} (DLQ, raw)`,
      );
    }
    await this.producer.send({
      topic: TOPIC_TRANSACTION_EVENTS_DLQ,
      messages: [{ key: key ?? undefined, value }],
    });
  }
```

## 4. Implementación en ai-service

Constantes y bucle de reintentos con **backoff** `300 * attempt` ms:

```14:15:services/ai-service/src/infrastructure/kafka/transaction-events.consumer.ts
const MAX_RETRIES = 3;
const delay = (ms: number) => new Promise((r) => setTimeout(r, ms));
```

Tras agotar intentos: log de error, `sendDlq`, y **persistencia en `processed_events`** para no reprocesar en bucle (comportamiento documentado en [Idempotencia §9](./3.Idempotencia.md#9-caso-límite-dlq-en-ai-service-sin-fila-de-explicación)):

```82:111:services/ai-service/src/infrastructure/kafka/transaction-events.consumer.ts
        let lastError: unknown;
        for (let attempt = 1; attempt <= MAX_RETRIES; attempt++) {
          try {
            this.logger.log(
              `[EVENT-BUS] EXEC [ai-service] ${env.eventType} eventId=${env.eventId} intento=${attempt}/${MAX_RETRIES}`,
            );
            await this.applier.apply(env);
            this.logger.log(
              `[EVENT-BUS] DONE [ai-service] ${env.eventType} eventId=${env.eventId} (explicación persistida)`,
            );
            return;
          } catch (e) {
            lastError = e;
            this.logger.warn(
              `Process attempt ${attempt}/${MAX_RETRIES} failed for ${env.eventId}: ${e}`,
            );
            if (attempt < MAX_RETRIES) {
              await delay(300 * attempt);
            }
          }
        }

        this.logger.error(
          `Moving event ${env.eventId} to DLQ after ${MAX_RETRIES} failures: ${lastError}`,
        );
        await this.kafkaService.sendDlq(env.eventId, raw);
        await this.processed.save({
          eventId: env.eventId,
          processedAt: new Date(),
        });
```

**Idempotencia previa:** si el `eventId` ya está en `processed_events`, el consumidor sale antes del bucle (evita llamadas redundantes al LLM).

## 5. accounts-service y transactions-service: sin DLQ en consumidor

Los consumidores de **accounts** y **transactions** **no** implementan hoy un topic DLQ ni reintentos explícitos en bucle. Ejemplo **accounts** (`TransactionEventsConsumer`): ante error en `eachMessage`, se registra y el offset sigue avanzando según la política del cliente (riesgo de **pérdida de procesamiento** si no se compensa fuera de este código):

```65:67:services/accounts-service/src/infrastructure/kafka/transaction-events.consumer.ts
        } catch (e) {
          this.logger.error(`Consumer error: ${e}`);
        }
```

Esto es una **decisión de simplicidad** del challenge: el sistema prioriza flujo feliz + idempotencia; la **observabilidad** y el **DLQ** están modelados con más detalle en **ai-service**, donde el coste del fallo (LLM) justifica reintentos y DLQ.

## 6. Errores en el publicador outbox

En `OutboxPublisherService`, si `kafka.send` falla para una fila, se hace **log** y la fila permanece `published: false` para el **siguiente** ciclo (reintento implícito):

```32:40:services/transactions-service/src/infrastructure/kafka/outbox-publisher.service.ts
    for (const row of batch) {
      try {
        await this.kafka.send(row.topic, row.partitionKey, row.payload);
        row.published = true;
        row.publishedAt = new Date();
        await this.outbox.save(row);
      } catch (e) {
        this.logger.warn(`Failed to publish outbox ${row.id}: ${e}`);
      }
    }
```

No hay DLQ de outbox: el registro queda en tabla hasta publicar.

## 7. Tabla comparativa

| Componente | Reintentos | DLQ | Notas |
|------------|------------|-----|--------|
| **ai-service** consumidor | Sí (3, backoff) | Sí (`transaction-events-dlq`) | Marca `processed` tras DLQ |
| **accounts-service** consumidor | No explícito | No | Error → log |
| **transactions-service** consumidores | Similar a accounts | No | Revisar cada consumer |
| **OutboxPublisher** (accounts / transactions) | Sí (siguiente tick) | No | Fila queda pendiente |

---

## 8. Buenas prácticas

1. **Acotar reintentos** y usar **backoff** para no amplificar incidentes.
2. **Correlación en logs:** `eventId` y `eventType` (ya presentes en ai-service).
3. **DLQ con payload original** para poder reprocesar o auditar.
4. Si se añade DLQ a accounts/transactions, alinear con **idempotencia** y política de offset (commit solo tras éxito o usar transaccional consumer si el stack lo permite).

## 9. Glosario

| Término | Definición breve |
|---------|------------------|
| **Retry** | Reejecutar el manejo del mismo mensaje tras un fallo transitorio |
| **DLQ** | Topic de mensajes “muertos” tras agotar reintentos |
| **Backoff** | Espera creciente entre intentos |
