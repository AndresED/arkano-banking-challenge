## 1. Un servicio = una aplicación Nest

Cada carpeta bajo `services/` es un **proyecto Node independiente** con:

- `src/main.ts` — punto de entrada.
- `src/app.module.ts` — módulo raíz.
- `src/modules/...` — funcionalidad de negocio (p. ej. `accounts`, `transactions`, `explanations`).

No se usa el transporte **TCP/RabbitMQ** built-in de `@nestjs/microservices` para el core: la comunicación entre procesos es **HTTP + Kafka (kafkajs)**.

---

## 2. AppModule y configuración

Patrón recurrente:

- **`ConfigModule.forRoot({ isGlobal: true })`** — variables de entorno (`.env`).
- **`TypeOrmModule.forRootAsync`** — conexión Postgres vía `DATABASE_URL`.

Ejemplo **accounts-service** (igual filosofía en otros servicios):

```6:21:services/accounts-service/src/app.module.ts
@Module({
  imports: [
    ConfigModule.forRoot({ isGlobal: true }),
    TypeOrmModule.forRootAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (config: ConfigService) => ({
        type: 'postgres' as const,
        url: config.getOrThrow<string>('DATABASE_URL'),
        autoLoadEntities: true,
        synchronize: true,
      }),
    }),
    AccountsModule,
  ],
})
export class AppModule {}
```

## 3. Feature modules (dominio)

El dominio “cuentas” vive en `AccountsModule`, que importa:

- **`CqrsModule`** — buses de comandos y queries.
- **`TypeOrmModule.forFeature([...entities])`** — repositorios inyectables.
- **`controllers`** — capa HTTP.
- **`providers`** — handlers + servicios Kafka/outbox/appliers.

Ver listado completo en [Hexagonal §3](./8.%20Hexagonal%20architecture.md#3-mapeo-a-arkano-accounts-service).

---

## 4. Inyección de dependencias

Nest resuelve constructores con metadatos:

- `@InjectRepository(Entity)` en handlers y servicios.
- Servicios `KafkaService`, `OutboxPublisherService` registrados como `providers`.

Los consumidores Kafka implementan `OnModuleInit` y arrancan `consumer.run` al inicializar el módulo.

## 5. Bootstrap: HTTP global

`main.ts` crea la app, aplica **ValidationPipe** (whitelist, forbid non whitelisted), interceptor de transformación de respuestas y filtro global de excepciones:

```7:20:services/accounts-service/src/main.ts
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,
      forbidNonWhitelisted: true,
      transform: true,
    }),
  );
  app.useGlobalInterceptors(new TransformInterceptor());
  app.useGlobalFilters(new AllExceptionsFilter());
  const port = process.env.PORT ?? 3001;
  await app.listen(port);
}
bootstrap();
```

Esto unifica validación de DTOs en los controladores sin lógica repetida en cada ruta.

## 6. Kafka como infraestructura en Nest

- **`KafkaService`:** encapsula cliente `kafkajs`, productor y (en ai) creación de topics al iniciar.
- **Consumidores:** clases `@Injectable()` registradas en el módulo; al iniciar se suscriben a topics.
- **Outbox:** `OutboxPublisherService` usa `OnModuleInit` + `setInterval` para flush periódico (ver [Outbox](./4.%20Outbox%20Pattern.md)).

---

## 7. “Microservices” de Nest vs este repo

Nest ofrece el paquete `@nestjs/microservices` con **patrones** y **transports** (Redis, NATS, gRPC, …). **Arkano no depende de ese paquete** para el bus principal: los “microservicios” son **tres procesos HTTP** que comparten **Kafka** como cola. Sigue siendo arquitectura de microservicios a nivel de **despliegue y datos**; solo el **mecanismo Nest-specific** de microservicios no es el elegido aquí.

## 8. Glosario

| Término | En Arkano |
|---------|-----------|
| **Module** | Unidad de composición (`AppModule`, `AccountsModule`, …) |
| **Provider** | Servicio inyectable (handlers, Kafka, publishers) |
| **Lifecycle hook** | `OnModuleInit` / `OnModuleDestroy` para Kafka |
