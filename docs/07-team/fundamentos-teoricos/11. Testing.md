## 1. Stack y convención de archivos

- **Jest** es el runner habitual en proyectos Nest (`*.spec.ts` junto al código fuente o en rutas espejo).
- Los archivos de prueba viven **cerca** del módulo probado: p. ej. `transaction-completed-applier.service.spec.ts` junto al servicio en `infrastructure/kafka/`.

Búsqueda rápida en el repo: `services/**/**/*.spec.ts` (accounts, transactions, ai).

---

## 2. Qué se testea en Arkano

| Área | Ejemplos de archivos |
|------|----------------------|
| **Kafka / appliers** | `transaction-completed-applier.service.spec.ts`, `transaction-execute.service.spec.ts`, `account-event-applier.service.spec.ts`, `ai-transaction-event-applier.service.spec.ts` |
| **CQRS handlers** | `request-transaction.handler.spec.ts`, `create-account.handler.spec.ts`, `get-account-by-id.handler.spec.ts`, … |
| **Dominio común** | `event-envelope.spec.ts` (por servicio) |
| **LLM / ai** | `llm-orchestrator.service.spec.ts`, `ollama-llm.service.spec.ts`, `mock-llm.service.spec.ts` |

La documentación de idempotencia ya lista varios de estos como “documentación viva” del comportamiento esperado ([Idempotencia §11](./3.Idempotencia.md#11-pruebas-unitarias-que-documentan-el-comportamiento)).

## 3. Ejemplo: applier de TransactionCompleted

El spec de **`TransactionCompletedApplierService`** valida reglas de negocio e idempotencia sin levantar Kafka real: se usa base en memoria o mocks de TypeORM según el archivo (abrir `transaction-completed-applier.service.spec.ts` para casos concretos como cuenta desconocida, saldo negativo, deduplicación por `eventId` y por pata).

**Ubicación:** `services/accounts-service/src/infrastructure/kafka/transaction-completed-applier.service.spec.ts`

---

## 4. Ejemplo: handler de comando

**`RequestTransactionHandler`** se prueba con transacciones y repositorios mockeados o DataSource de prueba para asegurar que:

- validaciones de DTO (deposit/withdrawal/transfer) disparan excepciones correctas;
- se persiste transacción `pending` y fila outbox con `TransactionRequested`.

**Ubicación:** `services/transactions-service/src/modules/transactions/application/commands/request-transaction.handler.spec.ts`

---

## 5. Ejemplo: envelope y parseo

Los tests de **`parseEnvelope`** protegen el contrato mínimo del bus: sin `eventId` / `eventType` / `source`, debe lanzar error. Eso alinea consumidores y productores.

**Ubicación:** p. ej. `services/accounts-service/src/common/events/event-envelope.spec.ts`

## 6. Ejecutar tests desde la raíz o por servicio

Desde cada servicio (patrón típico Nest):

```bash
cd services/accounts-service && npm test
```

Si el monorepo define scripts en la raíz, usar el del `package.json` raíz (si existe `npm run test --workspaces` o similar). **Ejecutar en tu entorno** el comando que tenga el `package.json` del workspace.

---

## 7. Huecos habituales y cómo ampliar

- **Consumidores Kafka** puros (`TransactionEventsConsumer`) suelen ser más pesados de testear en integración; Arkano concentra lógica en **appliers** testeables.
- **Pruebas e2e** con Kafka y Postgres reales son opcionales para CI; el challenge prioriza **unitarios** sobre servicios extraídos.
- Añadir tests cuando se cambie el contrato de eventos o la política de retry/DLQ.

|---------|-----|
