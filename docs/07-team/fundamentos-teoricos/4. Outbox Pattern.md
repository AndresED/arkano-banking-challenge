## 1. Problema: doble escritura

Actualizar **Postgres** y publicar en **Kafka** son dos sistemas distintos. Si primero confirmas la BD y luego falla el `send` al broker, quedas con **estado interno consistente pero sin evento** (otros servicios no se enteran). Si publicas primero y falla el commit, puedes tener **evento huérfano**. El outbox evita depender de un “commit distribuido” manual inconsistente: **persistes la intención de publicar en la misma transacción** que los datos de negocio.

---

## 2. Solución: outbox transaccional

1. En una **única transacción SQL**: guardas filas de negocio **y** una fila en `outbox_events` con el payload serializado (p. ej. JSON del envelope).
2. Tras el commit, un **proceso separado** (aquí un `setInterval` en Nest) lee filas `published = false`, envía a Kafka y marca `published = true`.

Así, **o** commitas negocio + outbox juntos, **o** ninguno; nunca “solo negocio” sin registro recuperable del evento (salvo bugs o rollbacks parciales fuera del alcance del patrón).

## 3. Esquema `outbox_events` en Arkano

La entidad es equivalente en **accounts-service** y **transactions-service** (cada uno su base de datos):

```8:37:services/transactions-service/src/infrastructure/persistence/outbox-event.orm-entity.ts
@Entity('outbox_events')
export class OutboxEventOrmEntity {
  @PrimaryGeneratedColumn('uuid')
  id!: string;

  @Column({ name: 'topic', type: 'varchar', length: 128 })
  topic!: string;

  @Column({
    name: 'partition_key',
    type: 'varchar',
    length: 128,
    nullable: true,
  })
  partitionKey!: string | null;

  @Column({ name: 'event_type', type: 'varchar', length: 128 })
  eventType!: string;

  @Column({ type: 'text' })
  payload!: string;

  @CreateDateColumn({ name: 'created_at' })
  createdAt!: Date;

  @Column({ name: 'published', type: 'boolean', default: false })
  published!: boolean;

  @Column({ name: 'published_at', type: 'timestamptz', nullable: true })
  publishedAt!: Date | null;
}
```

- **`topic` / `partition_key`:** destino Kafka y clave de partición (orden por cuenta o transacción según el caso).
- **`payload`:** cadena JSON del **envelope** completo (`eventId`, `eventType`, `payload` de negocio, etc.).

        eventId: randomUUID(),

          accountId,
          clientId: command.dto.clientId,
          balance: 0,
        },

      await outboxRepo.save({
        topic: TOPIC_ACCOUNT_EVENTS,

        eventType: 'AccountCreated',

```

  private async enqueueCompleted(
    outboxRepo: Repository<OutboxEventOrmEntity>,
    transactionId: string,
    accountId: string,
    amount: number,
  ): Promise<void> {
    const envelope: EventEnvelope = {
      eventId: randomUUID(),
      eventType: 'TransactionCompleted',
      source: 'transactions-service',
      occurredAt: new Date().toISOString(),
      version: EVENT_VERSION,
      payload: { transactionId, amount, accountId },

    await outboxRepo.save({
      topic: TOPIC_TRANSACTION_EVENTS,
      partitionKey: accountId,
      eventType: 'TransactionCompleted',
      payload: JSON.stringify(envelope),
      published: false,
    });
  }
```

En **accounts-service**, al aplicar `TransactionCompleted`, el applier inserta outbox para `BalanceUpdated` (mismo patrón transaccional dentro de `TransactionCompletedApplierService`).

## 5. Publicador en segundo plano

`OutboxPublisherService` arranca un **intervalo** (500 ms), toma un lote de hasta 50 filas no publicadas, envía con `KafkaService.send` y actualiza `published` / `publishedAt`:

```18:41:services/accounts-service/src/infrastructure/kafka/outbox-publisher.service.ts
  onModuleInit(): void {
    this.timer = setInterval(() => {
      void this.flushOnce().catch((e) =>
        this.logger.error(`Outbox flush failed: ${e}`),
      );
    }, 500);
  }

  async flushOnce(): Promise<void> {
    const batch = await this.outbox.find({
      where: { published: false },
      order: { createdAt: 'ASC' },
      take: 50,
    });
    for (const row of batch) {
      try {
        await this.kafka.send(row.topic, row.partitionKey, row.payload);
        row.published = true;
        row.publishedAt = new Date();
        await this.outbox.save(row);
      } catch (e) {
        this.logger.warn(`Failed to publish outbox ${row.id}: ${e}`);
      }
    }
  }
```

La implementación es **duplicada** (misma lógica) en `transactions-service` para mantener cada servicio autocontenido.

## 6. Dónde se usa en cada servicio

| Servicio | Tabla outbox | Eventos típicos escritos en outbox |
|----------|--------------|-------------------------------------|
| **accounts-service** | `outbox_events` | `AccountCreated`, `BalanceUpdated` (vía applier) |
| **transactions-service** | `outbox_events` | `TransactionRequested`, `TransactionCompleted`, `TransactionRejected` |
| **ai-service** | — | No usa outbox para dominio propio; consume y opcionalmente publica DLQ con productor directo |

---

## 7. Semántica de entrega y límites del diseño

- **Entre commit y Kafka:** hay **latencia** (hasta el siguiente tick del intervalo). Es coherente con “eventual” en el bus.
- **Fallo al publicar:** la fila queda `published = false`; el siguiente `flushOnce` reintenta (no es exactly-once en Kafka, pero el mensaje lleva `eventId` y los consumidores son idempotentes; ver [Idempotencia](./3.Idempotencia.md)).
- **Marcar `published` tras `send`:** si el proceso muere entre `send` exitoso y `save`, podría **republicarse** el mismo JSON (mismo contenido, mismo riesgo at-least-once que ya asume el sistema).
- **ai-service y DLQ:** `sendDlq` no pasa por outbox; es un canal de error operativo, no de negocio autoritativo.

## 8. Glosario

| Término | Significado en Arkano |
|---------|------------------------|
| **Outbox** | Tabla `outbox_events` + publicador periódico |
| **Envelope** | JSON en `payload` con `eventId`, `eventType`, `source`, etc. |
