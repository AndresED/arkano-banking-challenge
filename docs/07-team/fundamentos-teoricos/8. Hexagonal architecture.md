## 1. Idea central: dominio en el centro

La **arquitectura hexagonal** (Alistair Cockburn) coloca las **reglas de negocio** en el núcleo, sin depender de HTTP, Kafka ni SQL. El mundo exterior se conecta mediante **puertos** (interfaces que el dominio invoca o implementa) y **adaptadores** (implementaciones concretas: Nest controller, repositorio TypeORM, cliente Kafka).

El objetivo es **invertir dependencias**: el dominio no importa frameworks; los adaptadores sí conocen el dominio/DTOs.

---

## 2. Puertos y adaptadores en teoría

- **Puerto de entrada (driving):** API que expone casos de uso (REST, CLI, cola).
- **Puerto de salida (driven):** abstracción que el caso de uso necesita (persistencia, reloj, notificaciones).
- **Adaptador:** traduce entre el mundo externo y el puerto (JSON → comando, entidad → fila SQL).

---

## 3. Mapeo a Arkano (accounts-service)

El módulo de cuentas organiza el código en capas reconocibles:

- **`application/commands` y `application/queries`:** casos de uso invocados por el exterior (CQRS).
- **`application/dtos`:** validación y forma de entrada HTTP.
- **`infrastructure/adapters/in/rest`:** **adaptador de entrada** HTTP.
- **`infrastructure/persistence`:** **adaptador de salida** hacia Postgres (entidades ORM).
- **`infrastructure/kafka`:** adaptadores de **entrada** (consumidores) y **salida** (productor / outbox publisher).

El registro en `AccountsModule` conecta piezas:

```24:42:services/accounts-service/src/modules/accounts/accounts.module.ts
@Module({
  imports: [
    CqrsModule,
    TypeOrmModule.forFeature([
      ClientOrmEntity,
      AccountOrmEntity,
      OutboxEventOrmEntity,
      ProcessedEventOrmEntity,
      AppliedTransactionLegOrmEntity,
    ]),
  ],
  controllers: [AccountsController],
  providers: [
    ...Handlers,
    KafkaService,
    OutboxPublisherService,
    TransactionCompletedApplierService,
    TransactionEventsConsumer,
  ],
})
export class AccountsModule {}
```

Los **handlers** son el “caso de uso” aplicacional; el **controller** solo despacha al bus CQRS.

## 4. Adaptadores de entrada: REST y Kafka

**REST** (`AccountsController`): no contiene reglas de negocio complejas; delega en `CommandBus` / `QueryBus`:

```17:38:services/accounts-service/src/modules/accounts/infrastructure/adapters/in/rest/accounts.controller.ts
@Controller()
export class AccountsController {
  constructor(
    private readonly commandBus: CommandBus,
    private readonly queryBus: QueryBus,
  ) {}

  @Post('clients')
  @HttpCode(201)
  createClient(@Body() dto: CreateClientDto) {
    return this.commandBus.execute(new CreateClientCommand(dto));
  }

  @Post('accounts')
  @HttpCode(201)
  createAccount(@Body() dto: CreateAccountDto) {
    return this.commandBus.execute(new CreateAccountCommand(dto));
  }

  @Get('accounts/:id')
  getAccount(@Param('id', ParseUUIDPipe) id: string) {
    return this.queryBus.execute(new GetAccountByIdQuery(id));
  }
}
```

**Kafka** (`TransactionEventsConsumer`): adaptador de entrada que parsea el envelope y llama a `TransactionCompletedApplierService` (lógica de aplicación/infraestructura acoplada a TypeORM según el diseño actual).

## 5. Adaptadores de salida: persistencia y broker

- **TypeORM entities** en `infrastructure/persistence` representan el modelo físico (tablas).
- **`KafkaService.send`** y **`OutboxPublisherService`** publican hacia el broker sin que el controller HTTP conozca el detalle.

El **outbox** actúa como cola de salida **persistente** entre el commit transaccional y Kafka (ver [Outbox](./4.%20Outbox%20Pattern.md)).

---

## 6. Qué tan “puro” es el hexágono aquí

Arkano es un **proyecto educativo / challenge**: los handlers usan **TypeORM y `DataSource` directamente**, no interfaces `AccountRepository` en puertos explícitos. Aun así:

- La **separación física** `application` vs `infrastructure` vs `adapters/in/rest` sigue la **intención** hexagonal.
- Extraer **appliers** y **consumers** a servicios dedicados (`TransactionCompletedApplierService`) acerca el diseño a **testabilidad** y sustitución de adaptadores.

Un paso más “puro” sería definir puertos TypeScript e inyectar implementaciones en los handlers (no requerido por el alcance actual del repo).
